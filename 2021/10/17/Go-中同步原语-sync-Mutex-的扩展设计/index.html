Go 中同步原语 sync.Mutex 的扩展设计

def410
2021-10-17
Go, 并发编程

0x00 - 为什么需要互斥锁？

互斥锁是一种同步工具，在 Go 中，它可以保证同一时刻只有一个 Goroutine 进入临界区，访问共享资源时不被其他 Goroutine 打扰。

Go 语言标准库中提供了 sync.Mutex 这个同步原语，实现了互斥锁的功能，使用起来也非常简单，用 Lock() 和 TryLock() 两个方法就可以完成“上锁”和“解锁”的操作。同时，sync 包中还实现了读写锁 sync.RWMutex，对共享资源的读操作和写操作区别对待，在有大量读操作和少量写操作的并发场景下，读写锁的性能明显比互斥锁更高。

0x01 - sync.Mutex 的不足之处

sync.Mutex 这个类型是“开箱即用”的，使用起来非常简单，当然也有它的不足之处，比如：

  1. 在某些时候我们需要使用可重入锁，而 sync.Mutex 没有实现可重入的功能。
  2. sync.Mutex 这个结构体类型的内部字段中没有记录持有锁的 Goroutine 的 ID ，因此 Groutine A 上的锁可能被 Goroutine B 解开，在一些场景下，我们不希望这样的事情发生。
  3. 有时，我们在无法获取锁时，希望能立即得到一个获取失败的返回结果，然后去做别的事情，而不是一直阻塞，这就需要一个 TryLock 的方法来实现。
  4. sync.Mutex 没有暴露锁的状态信息，我们无法得知这个锁当前是否被持有，以及当前有多少 Goroutine 因这把锁而阻塞。

0x02 - 可重入锁的设计

需求

基于 sync.Mutex 设计一个可重入锁，同一个 Goroutine 可以重复上锁，并且锁只能被持有它的 Groutine 解开。

分析与设计

  1. 定义一个结构体类型，嵌入 sync.Mutex 并添加两个字段分别记录持有锁的 Goroutine ID（属主信息） 和重入次数。
  2. 为其实现 Lock 方法，如果是持有锁的 Goroutine 请求上锁，则记录重入次数并返回；否则调用嵌入的 Mutex 的 Lock 方法，成功上锁后记录属主信息，刷新重入次数并返回。
  3. 实现 UnLock 方法，若当前 Goroutine 未持有锁，则不允许解锁；否则将其重入次数 - 1，重入次数为 0 时则调用嵌入的 Mutex 的 UnLock 方法，彻底释放锁并清除属主信息。

0x03 - 获取互斥锁的状态

需求

我们希望能获取互斥锁当前的状态信息，包括：

  1. 锁是否被持有
  2. 是否处于饥饿状态
  3. 是否有被唤醒的 Goroutine
  4. 持有锁和因这把锁而阻塞的 Goroutine 数

分析与设计

要想获取这些信息，就不得不了解 sync.Mutex 内部字段的含义。

sync.Mutex 具有两个字段，其中 sema 字段表示信号量。state字段是一个复合字段，它使用不同的二进制位来表示多种信息，将其转为二进制，然后从右往左看，分别代表如下含义：

  1. 第 1位：标记锁是否被持有
  2. 第 2位：标记是否有唤醒的 groutine
  3. 第 3位：标记是否处于饥饿状态
  4. 剩余位：标记等待当前锁的 groutine数量

因此，如果能取到 state 字段的值，也就可以获取到锁的状态信息。

  • 取 state 字段的值 state 是 sync.Mutex 的第一个字段，一个 sync.Mutex 类型的值在内存中起始位置，就是指向的就是 sync.Mutex 的 state 字段的指针，这个指针可以通过 unsafe.Pointer() 方法来获取。
  • 判断锁的状态 取得 state 的值后，通过按位与运算，就可以知道 state 值转换成二进制后，每一位上是 0 还是 1，这样我们也就拿到了锁的状态信息！

0x04 - TryLock 方法的实现

需求

实现 TryLock 方法，如果成功获得锁则返回 true，否则立即返回 false ，而不是阻塞。

分析与设计

要实现 TryLock方法，我们还要利用刚刚分析过的 state 字段，通过 state 值来获取锁的状态，在不满足要求时返回 false，在满足要求时通过更改 state 值来对 Mutex 上锁。

  1. 调用 atomic 包中的 CAS 方法，若 state 值为 0 且成功赋值为 1 后，返回上锁成功。
  2. 否则获取 state 值，进行位运算，分析锁的状态，如果锁处于饥饿、唤醒、锁定状态，则返回 false。
  3. 如果未处于这三种状态，则表明这把锁未锁定，且有其他 Goroutine 在竞争，则使用 CAS 操作尝试修改 state 值，返回是否修改成功（是否成功上锁）。

0x05 - 总结

在某些并发场景中，我们需要可重入锁，以及具有 TryLock 方法的互斥锁，在 Go 标准库中的 sync.Mutex 类型基础上，我们可以很容易的进行扩展，这既能满足我们的需求，也能加深我们对 sync.Mutex 类型的理解。

锁是导致性能下降的万恶之源，通过获取锁的内部状态，我们可以在程序运行的某些时刻记录下“被某把锁阻塞的 Goroutine 数”，进而对程序性能问题进行分析、定位以及优化。
